## 個別仕様書評価

### atlas_v3
10 戦術の Protocol 統一は評価できるが、動的セレクターの判定ロジックがブラックボックス化しやすく、バグなし規律を「複雑性」で破壊する懸念がある。

### chronos_v3
MFFU ルールと TradersPost 連携の分離は正しいが、プロップ業者の急なルール変更（API 仕様変更）に対する動的な注入・切替機構が Interface レベルで未定義。

## common_v3 依存 DAG 評価
依存方向は正しい（Common が最下層）。ただし、Common が肥大化しており、Common の単体テストが完了するまで Atlas/Chronos の結合テストが一切行えない「ビッグバン統合」の罠に陥っている。

## Claude 起草者バイアスで見えない盲点
- 状態の永続化（State Persistence）の欠落: メモリ上の dataclass は定義されているが、再起動時にどこから復元するか（SQLite, JSON, Redis等）の Interface がない。これは『虚偽完了』の温床になる。
- 非同期設計の曖昧さ: 取引所 API は I/O 待ちが発生する。すべて同期（Sync）で書くと市場急変時に処理が詰まるが、非同期（Async）は非エンジニアのデバッグ難易度を爆上げする。この方針が未決定。
- 自己修復（Self-healing）の過信: 異常時に自動復旧する仕組みは、バグを『隠蔽』するリスクがある。バグなし規律を重視するなら、修復ではなく『安全な停止と証拠保全』に寄せるべき。
- テスト環境のコスト無視: カバレッジ 85% や Mutation Testing は、非エンジニアが維持するには CI/CD コスト（GitHub Actions 等）と学習コストが高すぎる。

## ゆうさくさん文脈での妥当性
月額ゼロ運用とバグなし規律には適合するが、保守運用フェーズで『なぜ動かないか』をゆうさくさんが判断するための『可読性の高いログ出力 Interface』が不足している。現状はエンジニア向けすぎる。

## Phase 2 着手前の必修対処 Top 3
### Priority 1: 永続化層（Storage Interface）の定義
根拠: idempotency や position_reconcile は、過去の状態を保存する DB が不可欠。この Interface を凍結しないと、Builder が勝手な形式で保存し、Atlas と Chronos で互換性がなくなる。

### Priority 2: 非同期（asyncio）採用可否の決定
根拠: Python での Bot 開発において、async の有無はコード構造を根本から変える。Phase 2 着手後に変更すると、全 Interface の書き直し（手戻り）が発生する。

### Priority 3: EICAS ログの物理フォーマット固定
根拠: ゆうさくさんが非エンジニアである以上、トラブル時に LLM にログを食わせて解析させる必要がある。LLM が解析しやすい構造化ログ（JSONL 等）の仕様を Interface に含めるべき。

## Gemini Flash 自身の限界（誠実開示）
