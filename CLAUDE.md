```
Bot（収益エンジン）

```

## 全担当者共通：思考の大前提（2026/04/11 ゆうさくさん明示）

### 戦略と戦術の違い
- **戦略**：目的を達成するための構造設計
- **戦術**：戦略を実行するための具体的手段
- 手段を最初から固定することは戦略の否定

### 戦術の結果が想定と違った場合のルール
テスト・実行結果が想定と異なる場合、戦術の微修正ではなく**戦略レベルからの練り直しを検討**することが大前提。
まず「その戦術を選んだプロセス自体に問題がないか」を疑う。

### 思考の優先順位
1. **目的思考**：最終目標（月300万円・2027年4月）から逆算して考える
2. **第一原理思考**：前提を疑い「本当にその手段が最善か」を毎回問い直す
3. **資金効率**：限られた資本で最大リターンを得る配分を考える
4. **機会損失の観点**：「今やらないことで失う機会」も必ずコストとして計上する

### 環境適応型の設計規律（最重要・2026/04/11確立）
**固定パラメータは環境適応ではない。Sora Labの宝であり規律。**

- 優秀なトレーダーは毎朝データを見て判断基準を調整する。Botもそうあるべき
- 「IVR < 30%でスキップ」は環境を見ているように見えるが、30%が固定である時点で半分しか適応できていない
- 正しい設計：環境データから閾値を動的に算出する仕組み
- 資金フェーズ（120万→400万→1億）でもパラメータは変わる。自動フェーズ移行を設計する
- 「一般的に言われている」「調査レポートにあった」は根拠として不十分
- 検証プロセス：実装 → バックテスト → デモで実データ検証 → 確定。飛ばさない

- 「なぜ今やらない？」に答えられないなら今やる

### 判断・実行の基準（2026/04/12確立）
**「実データが出てから」「検証結果を待ってから」は先延ばしの言い訳になりうる。**

| 根拠 | 対応 |
|---|---|
| 公式ドキュメント・APIメタデータで確認済み | **即実装・修正** |
| 複数の実践者事例で確認済み | **即実装・修正** |
| 実データ検証（追加確認） | 公式仕様がある場合は待つ理由にならない |
| 一般論・常識 | 根拠として不十分。調査してから判断 |

**実行前の自問（全担当必須）：**
1. 今できない技術的理由は何か？→ 答えられないなら今やる
2. 公式仕様がすでに確認済みか？→ YESなら実データを待つ必要はない
3. この待ち時間をゆうさくさんに説明できるか？→ NOなら今やる

### 調査品質基準（2026/04/12確立・全担当共通）
**「一般的に言われている」「調査レポートにあった」は根拠として不十分。（再掲・厳守）**

調査タスクでは必ず：
1. **公式仕様の直接確認** — 公式ドキュメント・API仕様・ソースコードを実際に読む
2. **複数の実践者事例** — GitHub・Reddit・ブログ・実コードで確認
3. **定量的な比較検討** — 「危険かも」ではなく数値で示す
4. **一般論で答えた場合は調査未完了として扱う**

### 実装前7ステッププロセス（2026/04/13確立・絶対規律）
**新手法のBot化は以下の全ステップを踏む。飛ばさない。**
1. **分解**: 手法の裁量要素を具体的に分解（何を見て何を判断しているか）
2. **データ特定**: その判断に使うデータはAPIで取得できるか
3. **ルール化**: 数値化・スコア化は可能か。先行事例はあるか
4. **課題洗い出し**: 技術・コスト・時間・リスクの各課題
5. **バックテスト**: ルール化した判断の精度検証
6. **精度検証**: 不十分なら何が足りないか
7. **実装**: 全ステップクリアして初めて実装
「Bot化不可能」と判断する前に、Step 1-4を完了すること。

## オーケストレーションルール

### 1. 調査→計画→実行の3フェーズ必須
破壊的操作・本番変更は必ず：
1. **調査**: 現状をread-onlyで確認
2. **計画**: Pushoverで計画報告・承認待ち
3. **実行**: 承認後のみ実行

### 2. VPS操作の安全規則
- OpenD/FutuOpenD の設定変更: 必ず計画をPushoverで報告してから
- 残り試行回数があるログイン操作: 1回失敗=即停止・報告
- サービス再起動は ops エージェントが担当
- 新規構築は builder エージェントが担当

### 3. 通知ルール
- 全タスク完了時: Pushoverで報告（必須）
- 障害・エラー時: Pushoverでpriority=1（緊急）
- 承認が必要な計画: Pushoverで送信してから待機

### 4. GitHub運用
TODOは`--label todo`、VPS実行は`--label hub-command`でIssue作成。hub_relay経由でVPS直接実行可。
